# Purpose

This document defines the documentation boundaries of
`docs/16-future-verticals/` — exactly what belongs inside this folder,
and what does not — before any Future Verticals documentation is
authored.

# Why This Repository Exists

`docs/16-future-verticals/` exists as a portfolio planning and
governance repository, not a business or platform domain: it holds
exploratory, non-binding documentation for potential future product
lines that are not yet committed to the platform roadmap, per the
approved Documentation Roadmap. It is not one of the five domains
`XP-0014` §9 anticipates a Domain Architecture document for, and it is
not a Digital Operating System governed by `XP-0030`. Unlike every other
folder covered by this standard, it is not working toward a single
future architecture document at all — per the approved Documentation
Roadmap, its documents are identified opportunistically, not planned, and
are never expected to reach `Approved` while still speculative.

# Repository Scope

This folder holds documentation that captures and evaluates uncommitted
future product-line ideas, and only that:

- Exploratory concepts and early-stage market/opportunity notes for
  potential future product lines, added opportunistically as identified.
- Speculative sketches explicitly marked non-binding, never asserted as
  architecture.
- Internal, non-authoritative planning material for evaluating whether a
  given exploratory idea warrants further attention.

# Out of Scope

- Platform-wide vision, purpose, or audience content — owned by
  `00-company/` and `01-platform/`.
- Core Platform capability content — owned by `02-core-platform/`.
- The constitutional meaning of "Module" itself — owned by `modules/`.
- The constitutional contract every Digital Operating System must
  satisfy — owned by `digital-operating-systems/`.
- The Future Vertical OS Domain Architecture already anticipated by
  `XP-0014` §9 as a committed roadmap item — that document is distinct
  from this folder's speculative content and is not hosted here.
- Any documentation asserting a committed capability, feature, workflow,
  integration, or implementation — once an idea is committed to the
  roadmap, its documentation belongs wherever the approved roadmap and
  authorization process place it, not here.
- Any other folder's own subject matter, including `03-business-os/`,
  `04-community-os/`, `05-life-os/`, `06-marketplace/`, `07-payments/`,
  `08-ai/`, `09-mobile/`, `10-devops/`, `11-security/`, `12-ui-ux/`,
  `13-api/`, `14-database/`, `15-development/`, or any other numbered
  folder — each owns its own documentation exclusively.
- Any documentation not yet authorized to be written under the Docs-First
  Rule and the Approval Gate.

# Architectural Dependencies

- `01-platform/` — the Platform Constitution, consulted only to check
  whether a speculative idea fits the platform's existing scope, per the
  approved Documentation Roadmap.

# Downstream Documentation

No folder in the approved Documentation Roadmap lists a dependency on
this one, and none appropriately could — this folder's content is
explicitly non-binding and never reaches `Approved`.

# Candidate Documents

No single top-level document is proposed. Per the approved Documentation
Roadmap, this folder's content is identified opportunistically rather
than planned toward a specific deliverable, and this document does not
introduce one.

# Current Status

`docs/16-future-verticals/` currently contains no authored
documentation. No document is planned; content is registered in
`DOCUMENT-INDEX.md` only if and when a specific exploratory idea is
identified, per the approved Documentation Roadmap's opportunistic
treatment of this folder.
