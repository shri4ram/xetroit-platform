# `modules/` — Module Architecture

**Status:** In Review

## Purpose

This folder holds the constitutional meaning of a **Module** in the
XETROIT Platform — what qualifies as one, how a proposed piece of work
is classified as one, how a Module is composed internally, and how it
evolves over time — shared across every Digital Operating System and,
where applicable, the Core Platform. It is an unnumbered, cross-cutting
top-level folder rather than a numbered domain folder, per
`00-governance/xp-0003-repository-standards.md`'s Folder Structure
Rules, since Module Architecture applies across every Digital Operating
System rather than belonging to any one of them.

| `xp-0029-module-lifecycle.md` | XP-0029 | Module Lifecycle |

`blueprints/module-architecture-blueprint.md` also lives in this folder,
in the `blueprints/` subfolder `xp-0003-repository-standards.md`
authorizes for non-authoritative Blueprint planning artifacts. It is a
planning reference for future Module Architecture documents, per
`00-governance/xp-0010-documentation-writing-standard.md` §3 — it
carries no authority of its own and is not one of the four documents
above.

## Documents That Belong Here

- The constitutional definition of a Module, its classification
  criteria, its internal composition principles, and its lifecycle —
  the four documents listed above.
- Non-authoritative Blueprint planning artifacts, in `blueprints/`, that
  precede and inform a future Module Architecture document.

A specific Module, or a specific Module Architecture document for one,
does not belong here — those are authored within the specific Digital
Operating System's own folder, or the relevant Core Platform folder,
once identified per `XP-0026` §13 and `XP-0014` §8.
